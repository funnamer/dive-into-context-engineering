⼀、什么是上下⽂⼯程
（1）定义
在传统的提示词⼯程中，上下⽂ C 只是⼀个单⼀的字符串（即 C=prompt）。⽽上下⽂⼯程将 C 重
新概念化为由多个信息组件（c1,c2,…,cn）动态组合⽽成的结构：
C=A(c1,c2,…,cn)
这⾥的 A 是⼀个“组装函数”，负责调度这些由不同函数获取、过滤和格式化的组件。这些组件主要
包括六个核⼼领域：
• cinstr (指令)：系统指令和规则。
• cknow (知识)：通过RAG（检索增强⽣成）或知识图谱获取的外部知识。
• ctools (⼯具)：可⽤外部⼯具的定义和签名（⽤于函数调⽤）。
• cmem (记忆)：历史交互中保留的持久化信息。
• cstate (状态)：⽤户、现实世界或多智能体系统的动态状态。
• cquery (查询)：⽤户当下的直接请求。
上下⽂⼯程本质上是⼀个寻找最优上下⽂⽣成函数集合的数学优化问题。我们的⽬标是找到⼀组理想
的函数 F（包括检索、选择、组装等），以最⼤化模型输出的预期质量（Reward），同时受限于模
型的最⼤上下⽂⻓度限制（∣C∣≤Lmax）：
⼀句话总结就是
构建有效的context结构，在上下⽂中找到 最⼩的⼀组⾼信息密度 token，同时最⼤化模型产⽣⽬
标结果的概率。

随着智能体的发展，prompt engineering这⼀范式正在逐步转变为context engineering
当⼀个 Agent 在循环（Loop）中运⾏，执⾏多轮推理和⻓周期任务时，它会不断⽣成新的数据（⽐
如⼯具调⽤结果、中间思考过程）。这些新数据必须被周期性地提炼和精简 (Cyclically refined)，否
则很快就会撑爆模型的注意⼒预算（Attention Budget）。提示词⼯程解决不了“历史消息越来越⻓”
或者“外部检索数据太多”的问题，⽽这正是上下⽂⼯程要解决的命题。
上下⽂⼯程不再是“如何写好⼀句话来引导AI”，⽽是构建⼀套严密的、模块化的系统架构。它通过科
学地调度指令、知识、⼯具、记忆和状态，让AI系统能够像⼈类⼀样，在复杂、模糊、动态且多模态
的现实世界中，做出准确的响应。
⼆、为什么需要上下⽂⼯程
（1）实际场景的问题
正如上⾯所提到的，现实⽣活中有相当⼀部分问题需要模型结合实时信息进⾏回答，需要提供额外信
息（这⾥可以是RAG、search⼯具等等）以供模型在新的知识上进⾏推理、回答，但是⼤量的信息会
导致模型的注意⼒漂移，如何有效的管理上下⽂，如何组织最少的上下⽂来让模型完成⽬标任务⼗分
重要，因此需要context-engineering。
（2）过⻓上下⽂带来的问题
⼤模型并不是全能的，它们在处理信息时⾯临着严苛的架构瓶颈。上下⽂⼯程是⽤来管理这些瓶颈的
必要⼿段：
• ⾃注意⼒机制的计算爆炸：LLM 基于 Transformer 架构，其核⼼的⾃注意⼒机制需要让每⼀个
Token 与其他所有 Token 建⽴联系。这会导致 n2 的计算复杂度和内存开销。随着上下⽂变⻓，
计算资源会呈⼆次⽅级消耗。
• 上下⽂腐败 (Context Rot) 与注意⼒预算：Anthropic 明确指出，LLM 就像⼈类的短期记忆⼀样，
拥有⼀个有限的“注意⼒预算（Attention Budget）”。在“⼤海捞针”测试中发现，随着 Token 数
量的增加，模型准确回忆信息的能⼒会下降（即上下⽂腐败）。每⼀个新加⼊的 Token 都在消耗
这个预算，导致注意⼒被“摊薄”。
• 训练数据的⻓短分布偏差：在模型的训练数据集中，短序列远⽐⻓序列常⻅。这意味着模型对于
“全局/⻓程依赖”的经验较少，缺乏专⻔处理超⻓上下⽂的参数。虽然有位置编码插值等技术来强
⾏扩展窗⼝，但这往往会牺牲模型对 Token 位置的精确理解，导致在⻓上下⽂中信息检索和逻辑
推理能⼒衰退。

（3）实际业务场景的问题
在商业和实际部署中，粗放的提示词和冗⻓的上下⽂会带来致命问题，必须通过⼯程化⼿段进⾏治
理：
• 缓解幻觉与不可靠性：如果上下⽂缺乏精⼼设计，LLM 极易出现幻觉（Hallucinations）、对输
⼊上下⽂不忠实、对微⼩变化过度敏感，或者给出“语法正确但缺乏语义深度”的废话。
• 降低延迟与 Token 成本：在商业应⽤中，每⼀次对话都在消耗 API 成本（按 Token 计费）。冗余
的上下⽂不仅拖慢了⽣成速度（Latency），还直接增加了财务成本。上下⽂⼯程通过智能内容
过滤和动态上下⽂优化，在不牺牲回答质量的前提下⼤幅降低 Token 消耗。
（4）多跳、复杂推理的问题
上下⽂⼯程不仅是为了“避坑”，更是为了“拔⾼”，它能让通⽤的 LLM 在垂直领域表现出专家级的能
⼒：
• 结构化推理与思维链：ToT（思维树）和规划（Planning）等结构化提示技术，上下⽂⼯程引导模
型拆解复杂问题，进⾏多步中间推理。
• ⽆需微调的灵活适配 (In-Context Learning)：上下⽂⼯程允许模型通过零样本（Zero-shot）、少
样本（Few-shot）演示和⻆⾊设定，直接适应新任务（如代码摘要、Bug 修复），⽽⽆需进⾏昂
贵的底层权重微调。并且精⼼设计的示例能让代码修复等特定任务的准确率飙升。
• 专业领域的性能⻜跃：在硬件设计、代码调试等⾼度专业化的领域，通过 RAG（检索增强⽣成）
和领域特定的上下⽂框架，可以直接将精确的外部知识注⼊模型，实现⾼达数倍的性能提升。
（5）核⼼理念：将上下⽂视为“稀缺资源”
传统的提示词⼯程往往把上下⽂当成⼀个“垃圾桶”，把所有可能相关的信息都塞进去。⽽上下⽂⼯
程的核⼼哲学发⽣了转变：
• 信息密度最⼤化：因为上下⽂是具有“边际收益递减”特性的稀缺资源，所以必须对其进⾏严格的
策展（Curate）。上下⽂⼯程利⽤压缩、选择和过滤机制，只保留信息密度最⾼的内容。
• 动态与⾃适应：未来的上下⽂⼯程将更加动态化，⽐如根据当前任务的需要，精准地在代码的语
法、语义、执⾏流和⽂档之间进⾏上下⽂类型的切换与调配。

（3）tool设计
设计原则
• Token-efficient 促进效率： ⼯具必须能够返回具有“Token 效率”的信息（避免冗⻓⽆效的信
息，如数百⾏⽆⽤的 JSON 嵌套），并⿎励智能体表现出⾼效的⾏为。通过精简输⼊和输出，只
保留推理必需的核⼼数据，从⽽保护⼤模型宝贵的上下⽂窗⼝不被冲刷。
• Clear ⽬的明确： ⼯具应该是⾃包含的（self-contained），具备极强的⽬的性。在微观设计上
必须做到逻辑⾃洽、⾼内聚，确保该⼯具只专注于完成⼀件特定的事情，降低模型的认知⻔槛。
• Non-overlapping 最⼩化功能重叠： 确保各个⼯具之间的功能互不重叠。如果连⼈类⼯程师都
⽆法明确在特定情况下该⽤哪个⼯具，AI 智能体同样⽆法做出正确判断。⼯具间的功能绝对隔离
能消除语义歧义，有效解决模型在判断时产⽣的“路由错误”或随机选择倾向。
• 易于被 LLM 理解： ⼯具的逻辑和定义必须契合⼤语⾔模型的理解⽅式。⼯具的命名、参数描述
和返回结果需要像“摘要”⼀样精准，且发挥⼤模型固有的语⾔理解优势，绝不能产⽣模棱两可的
理解。
• Robust 具备鲁棒性： ⼯具需要具备处理错误的能⼒，避免因单⼀⼯具调⽤失败（如 API 报错、
超时）导致整个流程崩溃或陷⼊死循环。优秀的⼯具应能在内部捕获异常，并将代码错误转化为
LLM 能看懂的“⾃然语⾔纠错建议”，引导其⾃⾏修正重试。
管理⽅式：
不要给 Agent 塞⼀个臃肿庞⼤的⼯具集。如果连⼈类⼯程师都不知道某⼀步该⽤哪个⼯具，Agent
也会陷⼊选择困难并浪费上下⽂。
• 最⼩可⾏集合（Minimal Viable Set）：精⼼策划并维持⼀个 Agent 所需的最少⼯具集合，确保
上下⽂信息丰富且紧凑（informative, yet tight）

记忆管理
记忆管理不仅仅是将信息存储起来，它还涉及信息的有效更新、检索和删除。⼀个⾼效的记忆管理系
统需要能够根据任务需求选择性地保存或忘记信息，以确保系统的性能和效率。
• 信息存储与更新：
在记忆系统中，信息是动态更新的。例如，短期记忆会根据新的对话内容不断被更新，⽽⻓期记
忆则根据⽤户的历史信息不断积累。为了避免记忆过载，系统会通过优先级算法来决定哪些信息
需要被保存，哪些可以被遗忘。这种信息选择与更新的过程确保了模型的效率和信息的相关性。
• 信息检索：
存储在记忆系统中的信息需要能够被快速检索出来。在⻓期记忆中，信息往往以向量的形式存
储，模型可以通过检索算法（如基于语义相似度的向量检索）从记忆中召回最相关的信息，补充
当前对话的上下⽂。这种信息检索不仅限于简单的关键词匹配，⽽是通过复杂的检索机制来选择
最相关、最具信息量的内容。
• 信息压缩：
记忆系统需要处理⼤量的信息，⽽过多的信息会导致存储空间的浪费并影响性能。因此，压缩技
术在记忆管理中扮演着重要⻆⾊。通过对存储的信息进⾏摘要、过滤或重组，系统能够将最重要
的部分保留，并去除冗余信息。例如，模型可以通过⽣成⼀个摘要来替代冗⻓的对话记录，或只
保留与当前任务⾼度相关的部分信息。
（5）知识注⼊通道（RAG）：
检索增强⽣成（Retrieval-Augmented Generation，RAG）是 上下⽂⼯程中最核⼼的架构之⼀。它
通过将 LLM 与外部知识库、向量搜索和结构化检索系统结合，使⽣成结果能实时接⼊外部信息来
源，从⽽解决 LLM 内部参数记忆的时效性和覆盖范围限制。
主要分为两类
模块化 RAG（Modular RAG）
模块化RAG的核⼼思想是将整个RAG过程分解为多个独⽴的、可插拔的模块。这些模块分别负责检
索、⽂档增强、⽣成等不同任务，每个模块可以根据需求进⾏独⽴的优化和替换。这种结构提供了极
⾼的灵活性，使得系统能够针对不同的任务需求进⾏定制化设计。
在模块化RAG中，每个模块的功能明确且独⽴。
• 检索模块负责从外部知识库中获取相关信息
• 查询重写模块则优化查询以提⾼检索的相关性

（1）什么是渐进式披露
“渐进式披露”并不是 AI 时代才出现的概念。它来⾃更早的交互设计领域。Jakob Nielsen 对它的经
典定义是：把⾼级的、低频的、次级的功能延后到第⼆层界⾯中，只先展示最重要、最常⽤的选项。
这样做的⽬的，是让系统更容易学习、更⾼效，也更不容易出错。NN/g 还特别指出，渐进式披露的
价值不只是“隐藏复杂性”，更是在向⽤户传达⼀种优先级：凡是默认出现在第⼀屏的信息，都是当
前最重要的信息。
如果把这个定义直接迁移到 LLM 系统⾥，意思就变成了：不要让模型在第⼀时间⾯对⼀个臃肿、混
杂、平铺直叙的上下⽂⼤包，⽽是要让它先看到最关键的线索，再根据任务推进逐层展开更深的内

明和⽂件”，其结构是完全同构的。不同的只是交互对象从“⼈类⽤户”变成了“作为推理主体的模
型”。

越多越好，⽽是⼀种稀缺的注意⼒资源分配问题。Anthropic 在《Effective context engineering for

（3）渐进式披露不是上下⽂⼯程的“旁⽀”，⽽是它的落地形式
很多⼈在第⼀次听到这两个词时，会把它们当成并列关系：上下⽂⼯程是⼀回事，渐进式披露是另⼀

上下⽂⼯程关⼼的是：什么信息该进⼊模型、何时进⼊、以什么形式进⼊、如何在⻓任务中保留与裁

楚：越来越多团队正在从“预先检索完再⼀次性喂给模型”，转向“保留轻量标识符，如⽂件路径、查

检索，⼜进⼀步“enables progressive disclosure”，也就是通过探索逐步发现相关上下⽂。

（4）它和 Just-in-Time 是什么关系
渐进式披露经常和 just-in-time ⼀起出现，但两者并不完全相同。可以把它们理解为两个互补维度。
Just-in-time 更偏时间维度。 它关⼼的是：信息什么时候取进来。不是预先把“所有可能有⽤”的内
容都塞进上下⽂，⽽是在推理进⾏到某⼀步时，才根据当前需要去检索、读取、执⾏。Anthropic 的
表述就是：agent 维护的是轻量级引⽤，⽽不是完整数据本体；通过⼯具在运⾏时动态加载内容。
Progressive disclosure 更偏结构维度。 它关⼼的是：信息按⼏层来揭示。先给⽬录级线索，再给
概要，再给细节，再给脚本或原始数据。它不是简单地“晚⼀点给”，⽽是要有明确的层级设计，让
系统知道第⼀层应该只承载⽅向感，第⼆层承载任务说明，第三层才承载执⾏细节。Anthropic 在
Skills 架构⾥将这种思想制度化为三层：元数据先加载；完整 只在相关时加载；更深层
SKILL.md
链接⽂件与资源只在需要时访问。
因此，可以把两者的关系概括为：JIT 解决“何时加载”，Progressive Disclosure 解决“按什么层级
加载”。 在⼀个优秀的 agent 系统中，它们往往同时成⽴。系统不会⼀次性预装全部内容，这是
JIT；系统也不会在第⼀次命中时就把所有相关⽂件全读进来，⽽是先读最上层说明，再按引⽤深
⼊，这是 Progressive Disclosure。两者结合，才真正构成现代上下⽂⼯程⾥的“按需上下⽂化”能
⼒。

（6）为什么这⽐“把所有相关知识都提前放进去”更优
直觉上，很多⼈会认为“我多给点背景，模型总归更懂”。这在简单问答⾥有时成⽴，但在复杂
Agent 场景⾥，常常会变成灾难。因为上下⽂⼀旦变⻓，问题就不再只是“够不够”，⽽是“有没有把
当前决策真正需要的信号淹没”。Anthropic 明确提醒，⻓上下⽂会带来 context pollution 和
relevance concerns，即便未来上下⽂窗⼝继续扩⼤，这类问题依然存在。

• 它降低了认知竞争。
模型不需要在⼀开始就同时权衡⼏⼗份说明、上百条规范、多个看似相关的⼯具接⼝，⽽是只处理当

要。这与 NN/g 对⼈类⽤户的分析⾼度⼀致：隐藏次级选项能够提升可学性、效率并减少错误率。
• 它更接近真实世界中的专业⼯作⽅式。
⼈类专家并不是把全部知识同时压在⼯作记忆⾥，⽽是依赖外部组织结构：⽬录、标签、书签、索
引、模板、速查表。Anthropic 也⽤类似的⽐喻说明，模型通过轻量引⽤在运⾏时取回信息，正如⼈
不会记住整个语料库，⽽是依靠⽂件系统、邮箱、书签等外部结构按需检索。
• 它允许系统把“信息”与“⾏动”解耦。
