# 五、Progressive disclosure（渐进式披露）

这部分内容虽然在authropic的blog中算作agentic search的一部分，但是这个设计哲学是贯穿整个上下文工程的，并且这个思想已经有很好的工程落地了。

在早期的 Prompt Engineering 范式中，一个常见的默认假设是：只要把背景、规则、示例和知识写得足够完整、足够具体，模型就更容易得到正确答案。但随着 AI 系统从单轮问答走向多轮推理、工具调用、长周期任务和多智能体协作，这种“一次性把所有信息塞进 prompt”的思路正在显露出局限。真正的问题往往不再是“给得不够多”，而是“给得不够准”“给得不够晚”。上下文窗口、注意力预算、推理延迟和 token 成本都是真实存在的工程约束；信息一旦过多，模型反而更容易被噪声淹没，把真正关键的线索埋在无关细节之中。Anthropic 在其 context engineering 文章中也明确指出，context engineering 的核心不是单纯写 prompt，而是持续策展和维护在每一步进入模型的最优 token 集合。  
因此，Progressive Disclosure（渐进式披露） 逐渐成为 Context Engineering 的一个关键原则。这个概念最早来自交互设计。Jakob Nielsen 对它的经典定义是：系统在初始界面中只显示少量最重要的选项，把高级、少用或更复杂的功能延迟到用户真正需要时再展开；在信息架构上，则由高层页面提供高层概念，由低层页面逐步补充细节。它的目标不是减少能力，而是减少认知负担，让复杂系统在不牺牲深度的情况下保持可理解、可操作。

迁移到 Context Engineering 中，渐进式披露的含义也随之发生变化：它不再只是“界面中的渐进展开”，而是上下文中的渐进展开。也就是说，模型在当前步骤不需要看到的信息，不应该提前占用上下文窗口；系统应先提供高层目标、结构索引、能力目录或局部线索，再让模型根据当下推理需要，逐步展开更细的说明、更重的文档、更长的证据和更复杂的工具资源。这里追求的不是“永远少给信息”，而是“按需给信息”；不是拒绝细节，而是把细节延迟到真正有用的时刻。  
Anthropic 在 2025 年的文章中指出，越来越多的 agent 不再在推理前预处理并载入全部相关数据，而是只维护轻量级标识符，如文件路径、存储查询和网页链接，并在运行时通过工具按需把数据拉入上下文。更重要的是，Anthropic 进一步明确写道：这种自主导航和检索机制会带来 progressive disclosure，agent 会通过探索逐步发现相关上下文，每一步得到的新线索都会影响下一步该去哪里看，从而以 layer by layer 的方式组装理解，只在工作记忆中保留当前真正必要的信息。

当前，Agent 系统的设计思想正深度汲取人类认知科学的灵感，其工作模式日益呈现出显著的“拟人化”趋势。在这一演进过程中，渐进式披露（Progressive Disclosure）的机制完美契合了人类的信息处理直觉。人类专家在面对长文档、复杂代码库或大型信息空间时，通常不会一开始就把所有原始材料逐字读完并装入短期记忆，而是先看目录、摘要、标题、文件名、时间戳、索引和搜索结果，再根据这些高层线索判断下一步应该深入哪一部分。Anthropic 也用非常接近这一点的方式类比人类认知：我们并不会记住整个语料库，而是依赖文件系统、邮箱、书签和其他外部索引体系，在需要时再取回相关信息。

这一思想在当前 agent 体系中已经有了非常清晰的工程化落地，其中一个最典型的例子就是 Agent Skills。OpenAI 的官方 Skills 文档明确指出，skills 使用 progressive disclosure 来管理上下文：系统启动时只把每个 skill 的 name、description 和 path 等元数据加入上下文，让模型先知道“有哪些能力存在”；模型根据这些元数据决定某个 skill 是否与当前任务相关；只有在真正需要调用该 skill 时，才会进一步读取完整的 SKILL.md；而更重的资源，如 scripts/、references/ 或 assets/，则继续延迟到执行阶段再加载。官方文档、API guide 和 cookbook 对这一点的描述是一致的。

从上下文工程视角看，Skill 的本质不再是“把一大段专家提示词硬塞进系统 prompt”，而是把专家知识打包成一个可路由、可激活、可按需展开的上下文模块。它天然具有三层结构：发现层只暴露 skill 的存在与适用范围；激活层载入完整操作说明；执行层再根据具体需要调用脚本和参考资料。换句话说，Skill 不是静态 prompt 的简单复用，而是基于渐进式披露构建的知识供给系统。

另一个基于该思想的工程落地便是PageIndex，它则是“文档证据如何渐进式加载”，PageIndex 会先把长文档转化为层级化的 tree / ToC index，再让 LLM 围绕这一结构进行 reasoning-based retrieval。它的典型流程不是直接把长文档或一批 chunk 全量送进上下文，而是先读取目录结构、理解文档组织方式；接着选择最可能相关的 section；再读取该节点内容；如果证据仍不足，就返回到结构层继续选择新的节点，直到信息足够后再生成答案。其文档甚至直接把这一流程写成“先读 ToC、再选 section、再抽取信息、若不足则继续循环”。  
这正是渐进式披露在检索层的体现。PageIndex 先让模型看到的是文档的高层结构，而不是文档的全部正文；只有当模型依据结构做出定位判断后，才去读取对应的局部内容；若局部内容不够，再按树继续扩展到邻近 section、子节点或引用节点。官方资料也明确强调，PageIndex 旨在模拟人类专家如何依靠目录、章节结构和交叉引用来定位答案，而不是单纯基于向量相似度匹配；其系统会生成一个保持原始逻辑流与组织结构的层级化 “table of contents” tree，并让 LLM 在这一结构上执行 tree search。

Agent Skills 和 PageIndex 虽然解决的是两类不同问题，却共享一个共同的底层设计哲学：不要把完整知识一次性塞入上下文，而是先给模型一个可导航、可判断、可路由的高层结构，再让模型在运行时逐步决定应该展开哪一部分。 在 Skills 中，被渐进式披露的是操作知识；在 PageIndex 中，被渐进式披露的是文档知识；他们面对的核心问题不再只是“往上下文里放什么”，而是：在什么时刻，以什么粒度，向哪个 agent 或哪个推理步骤披露什么。
